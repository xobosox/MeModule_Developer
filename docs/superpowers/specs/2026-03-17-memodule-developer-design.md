# MeModule Developer — Design Specification

**Date:** 2026-03-17
**Status:** Draft
**Author:** Tim + Claude

## 1. Overview

MeModule Developer is an AI-powered hosted platform for building ShareRing Me Modules. It enables the ShareRing community — primarily non-technical users and people with limited development experience — to plan, create, preview, and iterate on MeModules through natural language conversation with an AI assistant.

### 1.1 Goals

- Democratize MeModule creation beyond internal developers
- Provide a guided, AI-first experience where users describe what they want and the AI builds it
- Support visual planning (wireframes, mockups) before code generation
- Offer live preview of generated modules without requiring local tooling
- Include starter templates for common use cases
- Maintain a standardized tech stack for consistency and AI generation quality

### 1.2 V1 Scope

**In scope:**
- AI-powered module generation from natural language
- Visual planning with wireframes and mockups
- Live in-browser preview with mock ShareRing Me bridge
- Code editor for viewing and editing generated code
- Template library with 6 starter templates
- ShareRing Me app authentication
- Project save, version, and management

**Out of scope (future versions):**
- Built-in submission and review pipeline
- ShareRing Me bridge simulator (beyond mock data)
- Managed module hosting on subdomains
- Community template contributions
- Downloadable/local mode

### 1.3 Target Audience

**Primary:** ShareRing community members with no or limited development experience who want to create MeModules for the ecosystem.

**Secondary:** Developers who want a streamlined, AI-assisted workflow for MeModule development rather than setting up a local environment from scratch.

**Tiered experience:**
- Non-technical users interact primarily through chat and visual planning, with the AI handling all code generation
- Technical users can additionally view and edit generated code directly

## 2. Architecture

### 2.1 High-Level Architecture

```
┌─────────────────────────────────────────────┐
│  Browser (Client)                            │
│  ┌──────────┬───────────────────────────┐    │
│  │ Chat     │ Tabbed Panel              │    │
│  │ Panel    │ (Plan / Code / Preview)   │    │
│  └──────────┴───────────────────────────┘    │
└──────────────────┬──────────────────────────┘
                   │ WebSocket + REST
┌──────────────────▼──────────────────────────┐
│  Server (Node.js + Hono)                     │
│  ┌──────────┬───────────┬─────────────────┐  │
│  │ Auth     │ AI Engine │ Project         │  │
│  │ (SHR     │ (Claude   │ Storage         │  │
│  │  SSO)    │  API)     │ (DB + files)    │  │
│  └──────────┴───────────┴─────────────────┘  │
└──────────────────┬──────────────────────────┘
                   │
┌──────────────────▼──────────────────────────┐
│  External Services                           │
│  Claude API │ ShareRing Auth │ Template Repo │
└─────────────────────────────────────────────┘
```

### 2.2 Approach: Hybrid Client-Side Preview + Server AI

- **AI generation** runs on the server via the Claude API
- **Live preview** runs entirely in the browser using esbuild-wasm for transpilation
- **No server compute per user session** — the server only handles API calls, auth, and storage
- Pre-bundled dependencies and import maps eliminate the need for `npm install` in the browser
- This approach is lightweight (~2-3MB WASM), has good browser compatibility, and keeps infrastructure costs low
- Can be upgraded to WebContainers in the future if more advanced build capabilities are needed

### 2.3 Tech Stack

**Platform (this project):**
- Frontend: React + TypeScript + Vite + TailwindCSS
- Code Editor: Monaco Editor
- Preview Engine: esbuild-wasm + import maps + sandboxed iframe
- Server: Node.js + Hono
- Database: PostgreSQL
- Real-time: WebSocket (for AI streaming)
- AI: Claude API (Anthropic)
- Auth: ShareRing Me SSO

**Generated MeModules (standardized):**
- React 18 + TypeScript
- Vite (build tool)
- Zustand (state management)
- TailwindCSS (styling)
- React Router with hash routing
- ShareRing Me bridge helper

## 3. User Interface

### 3.1 Layout: Two-Panel with Tabs

The workspace uses a two-panel layout optimized for the non-technical target audience:

**Left Panel — AI Chat:**
- Persistent chat interface for conversing with the AI
- User describes what they want, asks for changes, gets guidance
- Streaming responses shown in real-time
- Full conversation history preserved per project

**Right Panel — Tabbed Content:**
- **Plan tab:** Visual wireframes, screen flow diagrams, and architecture overview generated by AI during planning phase
- **Code tab:** Monaco editor with file tree sidebar, syntax highlighting, multi-file tabs. Users can view and edit generated code.
- **Preview tab:** Live rendered preview of the MeModule in a mobile-frame mockup with mock bridge data

The tab system naturally guides users through the workflow: Plan → Code → Preview. The AI can auto-switch tabs (e.g., show Preview tab when wireframes are generated).

### 3.2 Dashboard

The landing page after login:
- "Create New Module" button (starts AI chat)
- "Start from Template" section with template gallery cards
- "My Projects" list with saved projects (name, status, last modified, thumbnail)

### 3.3 Template Gallery

Grid of template cards showing:
- Thumbnail/preview image
- Template name and description
- Category tag (loyalty, payments, events, etc.)
- "Use Template" button → creates project and opens workspace

## 4. User Flows

### 4.1 New Module from Scratch

1. User logs in via ShareRing Me → developer mode auto-enabled in app
2. Dashboard shows → user clicks "Create New Module"
3. Workspace opens with empty project and chat panel focused
4. User describes their idea in natural language
5. AI asks clarifying questions (one at a time)
6. AI generates wireframes → shown in Plan tab (auto-switches)
7. User reviews wireframes, requests changes via chat
8. Once plan approved, AI generates full module code
9. Code appears in Code tab, live preview in Preview tab
10. User iterates via chat ("change the color", "add a page") or edits code directly
11. When satisfied → downloads/exports the built module as a zip

### 4.2 Start from Template

1. User browses template gallery on dashboard
2. Selects a template → new project created with template file tree
3. Workspace opens with template code loaded and preview running
4. AI has context that this started from a template
5. User customizes via chat or code editor
6. Live preview updates as changes are made

### 4.3 Resume Existing Project

1. User opens a saved project from dashboard
2. Workspace loads with full file tree and chat history
3. User continues where they left off

## 5. Server Architecture

### 5.1 Data Model

```sql
users
  id              TEXT PRIMARY KEY  -- from ShareRing auth
  sharering_address TEXT NOT NULL
  developer_mode_enabled BOOLEAN DEFAULT true
  created_at      TIMESTAMP DEFAULT NOW()

projects
  id              UUID PRIMARY KEY
  user_id         TEXT REFERENCES users(id)
  name            TEXT NOT NULL
  description     TEXT
  template_id     UUID REFERENCES templates(id) NULL
  status          TEXT DEFAULT 'draft'  -- draft | building | ready
  file_tree       JSONB NOT NULL        -- virtual filesystem
  created_at      TIMESTAMP DEFAULT NOW()
  updated_at      TIMESTAMP DEFAULT NOW()

conversations
  id              UUID PRIMARY KEY
  project_id      UUID REFERENCES projects(id)
  messages        JSONB NOT NULL        -- chat history array
  updated_at      TIMESTAMP DEFAULT NOW()

templates
  id              UUID PRIMARY KEY
  name            TEXT NOT NULL
  description     TEXT NOT NULL
  category        TEXT NOT NULL          -- loyalty, payments, events, surveys, etc.
  thumbnail_url   TEXT
  file_tree       JSONB NOT NULL        -- starter files
  tags            TEXT[]
```

### 5.2 File Storage Strategy

The `file_tree` JSONB column stores the complete virtual filesystem as a flat map of path → content:

```json
{
  "src/App.tsx": "import ...",
  "src/screens/Home.tsx": "export function Home() ...",
  "manifest.json": "{ \"version\": \"0.0.1\" ... }"
}
```

**Write strategy:** Individual file updates (from editor or AI) use PostgreSQL `jsonb_set()` to update a single key without rewriting the entire JSONB. This is atomic and handles concurrent updates safely.

**Concurrency:** During AI streaming, file updates are queued server-side and applied sequentially. If the user edits a file while the AI is streaming changes to a *different* file, both proceed independently. If the user edits a file the AI is currently writing, the user's edit takes priority — the AI's version for that file is discarded and the AI is informed of the user's change.

**Limits:** Maximum 100 files per project, 500KB per file, 10MB total per project.

### 5.3 API Endpoints

**Authentication:**
- `POST /auth/sharering` — Authenticate via ShareRing Me, create/update user, enable developer mode

**Projects:**
- `GET /projects` — List user's projects
- `POST /projects` — Create new project (blank or from template)
- `GET /projects/:id` — Get project details + file tree
- `PUT /projects/:id` — Update project metadata
- `DELETE /projects/:id` — Delete project

**Files:**
- `GET /projects/:id/files` — Get full file tree
- `GET /projects/:id/files/*path` — Get single file content
- `PUT /projects/:id/files/*path` — Update a single file (uses `jsonb_set()` on `file_tree`)
- `POST /projects/:id/files/*path` — Create a new file
- `DELETE /projects/:id/files/*path` — Delete a file

**AI Chat:**
- `WS /projects/:id/chat` — WebSocket for AI conversation + code generation streaming (see Section 6.5 for protocol)

**Templates:**
- `GET /templates` — List all templates (with optional category filter)
- `GET /templates/:id` — Get template details + file tree

**Export:**
- `POST /projects/:id/export` — Package project source code as downloadable zip

## 6. AI Engine

### 6.1 System Prompt Architecture

The AI uses a layered prompt system for high-quality MeModule generation:

**Layer 1 — Base Knowledge:**
- Complete ShareRing Me bridge API reference (all 64 events across 9 categories)
- Manifest configuration schema and rules
- Required patterns: hash routing, relative asset paths, `base: "./"` in Vite config
- Standardized stack conventions (React + TS + Vite + Zustand + TailwindCSS)
- Mobile-first design principles and safe area handling
- The `createShareRingMeBridge()` helper implementation

**Layer 2 — Template Context:**
- When starting from a template, the template's architecture and file structure are injected so the AI understands the starting point

**Layer 3 — Project Context:**
- Current project's complete file tree
- Conversation history (full chat context)

**Layer 4 — User Request:**
- The user's current message

### 6.2 AI Capabilities by Phase

| Phase | AI Action | Output |
|-------|-----------|--------|
| Planning | Ask clarifying questions | Chat messages |
| Planning | Generate wireframes | HTML wireframes → Preview tab |
| Planning | Propose screen flow | HTML diagram → Plan tab |
| Building | Generate full project scaffold | Complete file tree (manifest, router, store, screens, services) |
| Building | Generate individual components | Single file creation/update |
| Iterating | Modify existing code | Targeted file updates |
| Iterating | Refactor/restructure | Multi-file updates |
| Iterating | Explain code | Chat messages |

**Plan tab rendering:** The Plan tab renders AI-generated content as HTML. Screen flow diagrams, architecture overviews, and wireframes are all generated as styled HTML by the AI and rendered in a scrollable container. This keeps the rendering simple (no Mermaid/SVG parsing needed) and gives the AI full control over visual presentation. The AI uses inline CSS for layout (flexbox grids for screen flows, styled boxes for architecture diagrams).

### 6.3 Structured Output Format

The AI is called using Claude's **tool use** feature. The system prompt defines tools that map to output types:

```
Tools defined for the AI:
- chat(content: string)        — Send a message to the user
- write_file(path: string, content: string) — Create or update a project file
- show_preview(content: string) — Show HTML wireframe/mockup in preview
- show_plan(content: string)   — Show HTML plan/diagram in plan tab
```

The AI calls these tools in its response. The server processes each tool call and emits a corresponding typed JSON message over the WebSocket. This approach is robust because:
- Tool calls have structured parameters (no text parsing needed)
- Claude is designed to produce well-formed tool calls
- File content inside tool call parameters is cleanly separated from other output
- No risk of generated code containing delimiter strings

The client routes each message by type:
- `chat` → Chat panel (appended to current message)
- `file` → File tree update + code editor
- `preview` → Preview tab (auto-switches)
- `plan` → Plan tab (auto-switches)

**Streaming:** Claude's streaming API emits tool call arguments incrementally. The server buffers each tool call until complete, then emits the full message to the client. Chat tool calls are streamed token-by-token (partial delivery). File tool calls are buffered and delivered complete (since partial file content is not useful to the editor).

### 6.4 Conversation Size Management

Conversations will eventually exceed Claude's context window. The system manages this with a sliding window approach:
- The most recent N messages (e.g., last 20) are always included in full
- Older messages are summarized into a condensed context block by the AI before being evicted
- The current file tree is always included (it represents ground truth)
- If even the summary + recent messages + file tree exceeds limits, older summaries are dropped

### 6.5 WebSocket Chat Protocol

**Client → Server messages:**

```json
{"type": "user_message", "content": "Change the header color to blue"}
{"type": "cancel", "reason": "user_cancelled"}
{"type": "file_edited", "path": "src/App.tsx", "content": "..."}
```

The `file_edited` message is sent when the user edits a file in the code editor while the AI is streaming. The server:
1. Applies the user's edit to the project file tree immediately
2. If the AI is currently streaming and has not yet emitted a `file` message for the same path, no conflict
3. If the AI emits a `file` message for the same path after the user edit, the server discards the AI's version and injects the user's current content into the next AI request context

**Server → Client messages:**

All server messages include a sequential `seq` field for ordering and reconnection:

```json
{"seq": 1, "type": "connected", "projectId": "..."}
{"seq": 2, "type": "chat", "content": "partial text...", "streaming": true}
{"seq": 3, "type": "chat", "content": "full message", "streaming": false}
{"seq": 4, "type": "file", "path": "src/App.tsx", "content": "..."}
{"seq": 5, "type": "preview", "content": "<html>...</html>"}
{"seq": 6, "type": "plan", "content": "<div>...</div>"}
{"seq": 7, "type": "generation_complete"}
{"seq": 8, "type": "error", "code": "rate_limit", "message": "Too many requests, try again in 30s"}
```

Chat messages use `streaming: true` for partial content (token-by-token) and `streaming: false` for the final complete message. File, preview, and plan messages are always delivered complete (buffered from tool calls).

**Connection lifecycle:**
- Client connects on workspace open, authenticates via JWT query param
- Server sends `{"seq": 1, "type": "connected", "projectId": "..."}` on successful auth
- On disconnect, client reconnects with exponential backoff (1s, 2s, 4s, max 30s)
- On reconnect, client sends `{"type": "resume", "lastSeq": 5}` — server replays any messages with `seq > 5` from a server-side buffer (last 100 messages kept in memory per session)

### 6.6 Quality Guardrails

- `manifest.json` is always generated with correct schema
- Bridge helper code (`createShareRingMeBridge()`) is injected as a read-only project file, not regenerated each time
- Generated code is validated against MeModule constraints:
  - Hash routing (no path-based routing)
  - Relative asset paths
  - Correct Vite config (`base: "./"`)
  - Proper bridge event types (uppercase)
- AI is instructed to generate mobile-first responsive layouts
- Preview auto-refreshes when file changes are streamed

## 7. Client-Side Preview Engine

### 7.1 Transpilation

- **esbuild-wasm** runs in the browser to transpile TSX/TS to plain JS (~50ms per file)
- No bundling step — only transpilation. Module resolution handled by import maps.
- On file change, only the changed file is re-transpiled (hot refresh)

### 7.2 Pre-bundled Dependencies

The platform ships pre-bundled ESM versions of all allowed libraries:
- `react` + `react-dom`
- `react-router-dom`
- `zustand`
- ShareRing Me bridge helper

These are served as static assets and resolved via the iframe's import map.

### 7.3 Import Map Resolution

The preview iframe uses an [import map](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/script/type/importmap) to resolve:
- Bare imports (`react`, `zustand`, etc.) → pre-bundled ESM URLs
- Project-internal imports (`./screens/Home`) → blob URLs of transpiled code

### 7.4 Preview Iframe

A sandboxed iframe loads a generated `index.html` that:
- Includes the import map pointing to pre-bundled deps
- Includes TailwindCSS via the Tailwind Play CDN (for dev preview)
- Mounts the user's React app
- Wraps in a mobile phone frame mockup for realistic preview
- Provides a mock ShareRing Me bridge (see below)

### 7.5 Mock Bridge

The preview includes a simulated ShareRing Me bridge so modules are interactive without the real app:

| Event | Mock Response |
|-------|--------------|
| `WALLET_CURRENT_ACCOUNT` | `{ address: "shareledger1mock...", pubKey: "mock..." }` |
| `WALLET_BALANCE` | `[{ amount: "1000000000", denom: "nshr" }]` |
| `COMMON_APP_INFO` | `{ language: "en", darkMode: true, version: "1.0.0" }` |
| `COMMON_ASYNC_STORAGE` read/write | In-memory key-value store |
| `VAULT_DOCUMENTS` | Sample document list |
| `VAULT_EMAIL` | `"user@example.com"` |
| `CRYPTO_ENCRYPT/DECRYPT` | Pass-through (no real encryption) |

Users see realistic data in their preview. The mock bridge can be extended with custom test data in future versions.

### 7.6 Error Handling in Preview

The preview must handle errors gracefully since non-technical users will encounter them frequently:

**Syntax errors (transpilation failure):**
- esbuild-wasm returns the error with line/column info
- Preview shows a styled error overlay with the error message, file name, and line number
- The error overlay includes a "Fix with AI" button that sends the error to the chat, prompting the AI to fix it

**Runtime errors (React crashes):**
- A React error boundary wraps the entire preview app
- On crash, shows a friendly error screen with the error message
- "Fix with AI" button sends the stack trace to chat

**Missing imports (non-pre-bundled library):**
- Import map resolution fails with a clear error: "Library X is not available. Available libraries: react, zustand, ..."
- Shown in the error overlay with guidance

**All errors are non-destructive** — the last working preview state is preserved until the error is fixed. The user always has a working reference to go back to.

### 7.7 Limitations (acceptable for v1)

- Only pre-bundled dependencies available (no arbitrary `npm install`)
- TailwindCSS via Play CDN (not compiled; may cause brief flash of unstyled content on first render)
- Mock bridge only — real bridge testing requires the ShareRing Me app and developer mode
- Some edge cases in module resolution may not match production Vite builds

- All source files (screens, store, services, types)

## 9. Authentication

### 9.1 ShareRing Me Login

Users authenticate using their ShareRing Me app identity:
- Platform initiates auth flow via ShareRing SSO
- On successful auth, server receives ShareRing address and identity
- Server creates or updates user record
- JWT session token issued for subsequent API calls

### 9.2 Developer Mode Auto-Enable

When a user logs into MeModule Developer for the first time, developer mode is enabled for them in the ShareRing Me app:

**Integration mechanism (decision deferred — requires ShareRing platform team coordination):**
- **Option A: ShareRing Admin API** — If available, the server calls a ShareRing API endpoint to enable developer mode for the user's address. This is the cleanest approach.
- **Option B: Deep link / QR code** — The platform generates a deep link or QR code that the user opens in the ShareRing Me app, which triggers developer mode enablement. More manual but requires no backend API.
- **Option C: Manual with guidance** — The platform shows step-by-step instructions to enable developer mode in Settings → Developer Tool. Least friction to implement.

The exact mechanism depends on what APIs the ShareRing platform exposes. For v1, **Option C is the fallback** if no API is available, with Option A as the goal.

**Failure handling:** If auto-enablement fails (API error, timeout), the user is shown manual instructions as a fallback. The platform does not block usage — the user can still build modules, they just need to enable developer mode manually to test in the app.

## 10. Export & Deployment

### 10.1 V1: Export as Zip

Users can export their module as a zip file containing **source code only** (no pre-built dist):
- `src/` directory with all source files
- `package.json` with all dependencies listed
- `vite.config.ts` pre-configured with `base: "./"` and required polyfills
- `tsconfig.json`
- `manifest.json`
- `index.html` entry point
- `README.md` with setup, build, and deployment instructions

The zip is a fully self-contained project that can be:
- Opened in VS Code for further development
- Built locally with `npm install && npm run build`
- The resulting `dist/` folder can be hosted on any static hosting provider

**Why source-only:** The platform does not run server-side builds (no compute per session). A production Vite build requires a full Node.js environment, which users run locally. The README includes clear instructions for this.

### 10.2 Future: Managed Hosting + Submission

In later versions:
- One-click deployment to a managed subdomain
- Built-in submission to ShareRing review pipeline
- Review status tracking in the dashboard

## 11. Non-Functional Requirements

### 11.1 Performance
- Preview transpilation: < 100ms per file change
- AI response streaming: first token < 1s
- Dashboard load: < 2s
- Workspace load (with project): < 3s

### 11.2 Security
- All AI API keys server-side only (never exposed to client)
- ShareRing auth tokens validated on every request
- Project data isolated per user (no cross-user access)
- Preview iframe sandboxed (`allow-scripts` only, no `allow-same-origin`)
- Rate limiting on AI chat endpoints
- WebSocket auth via short-lived ticket (issued via REST, used once in WS handshake) to avoid JWT in query strings appearing in logs

### 11.3 Scalability
- Stateless server (horizontal scaling behind load balancer)
- AI streaming via WebSocket with connection pooling
- WebSocket reconnection replay buffer stored in Redis (shared across server instances) rather than in-memory
- File trees stored as JSONB (efficient for read-heavy workloads)
- CDN for static assets and pre-bundled dependencies
- Conversation storage: max 500 messages per conversation; older messages summarized and archived
